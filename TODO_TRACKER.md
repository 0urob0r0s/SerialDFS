## Discoveries & Deferred Items

| ID | Discovery | Impact | Deferred to |
|---|---|---|---|
| D1 | DOSBox-X bypasses INT 2Fh AH=11h for normal file ops. `src/dos/dos_files.cpp:494` dispatches every operation directly through its internal `Drives[]` C++ table; `CDSFLAG_NET\|PHY` in the CDS is ignored. | Phase 9/10 e2e tests of redirected file ops cannot run inside DOSBox-X. | **OBSOLETED 2026-04-26 by 86Box migration.** 86Box uses real BIOS + INT 2Fh dispatch. e2e_tsr.sh now exercises DIR through the redirector and PASSES 7/7. |
| D4 | TSR `AL_SPOPNFIL` handler bug, surfaced after 86Box migration removed D1. DOS COMMAND.COM's `COPY X:\src C:\dst` and `TYPE X:\file` issue INT 2Fh `AL_SPOPNFIL` (extended open) with action_code = 0x01 (hi=0, lo=1 = "fail if exists, create if not"). The handler at dos/src/serdfs.c originally sent `CMD_CREATE mode=0` to the daemon, which correctly returned ERR_EXISTS for an existing source. COPY/TYPE then bailed silently. | `e2e_copy.sh` and `fault_inject.sh` blocked; e2e_tsr.sh's TYPE + MKDIR steps blocked. | **FIXED 2026-04-27.** AL_SPOPNFIL now probes via CMD_OPEN whenever the action could legitimately open an existing file (hi_act∈{1,2} OR action 0x01). For action 0x01 specifically we treat the probe success as "opened existing" (CX=1), matching DOS COMMAND.COM's empirical expectation. e2e_tsr.sh: 9/9 PASS, e2e_copy.sh: 6/6 PASS, fault_inject.sh: 6/6 PASS. |
| D6 | CMD_READ in the daemon (`linux/serdfsd/fsops.py _read`) advanced the file pointer on every call. TSR retries on serial timeout sent the same READ request again, but the daemon served the *next* chunk instead of the same one — silently misordering the destination file by 512 bytes per retry. | `e2e_copy.sh` reported "1 file(s) copied" but extracted destination differed from source (~6 KB short on 64 KB transfer due to 12 retries during the test). | **FIXED 2026-04-27.** CMD_READ payload extended from `hid(1)+count(2 LE)` to `hid(1)+count(2 LE)+offset(4 LE)`. Daemon does `os.pread(fd, count, offset)` instead of `os.read(fd, count)` when the 7-byte form is received (3-byte legacy form preserved for SERTYPE.EXE et al.). TSR sends `sft->file_pos + totread` as the offset; retries are now idempotent. |
| D7 | (Original framing — superseded by D7' below.) `MEM /C` after `SERDFS install` was thought to hang DOS under 86Box+QEMU. | e2e_tsr.sh cannot include MEM /C in the same BAT as DIR/TYPE/MKDIR. | Replaced by D7' with a sharper bisect. |
| D7' | **2026-04-27 bisect with the new `86box-pcmd`.** The hang is NOT MEM /C — MEM /C completes successfully both standalone and inside a multi-line BAT after `SERDFS install`. The hang is triggered specifically by **a redirector dispatch (`DIR X:\` or any X: file op) that follows `MEM /C` in the same DOS session**. Tested matrix on 86box-cmd: `install + MEM /C + /U` works. `install + DIR X:\ + MEM /C + /U` works (DIR returns the daemon listing; MEM shows 11K resident; /U clean). `install + MEM /C + DIR X:\ + /U` HANGS — DOS doesn't return from the DIR call; the BAT never writes A:\DONE; 86box-cmd times out. The hang is order-dependent: MEM-then-DIR breaks; DIR-then-MEM works. Hypothesis (unverified): MEM /C's UMB/XMS walk or memory-summary computation dirties some piece of state (a DOS internal pointer? a CDS attribute? an MCB owner?) that the next INT 2Fh AH=11h dispatch trips over. The TSR's INT 2Fh handler probably never gets called — DOS's pre-dispatch validation likely loops or aborts before the call. | `e2e_tsr.sh` deliberately omits MEM /C from its BAT; running it in a separate cold boot (or via `86box-pcmd`) works fine. | Workaround in production. Real root-cause investigation would need an in-DOS debugger break on the DIR's INT 2Fh entry to confirm whether DOS even tries to call our handler — out of scope for v1. |
| D8 | `86box-pcmd` (the real one — COM2-based persistent DOS REPL via PCMDD.EXE) installs SerialDFS as TSR, /STATS multiplex responds (proves TSR resident), but `DIR X:\` issued in a subsequent `pcmd run` returns blank with `/STATS` showing TX:0 RX:0 — i.e., DOS-side never actually issues an INT 2Fh dispatch for the X: drive even though SERDFS is loaded. | TSR-persistence testing under pcmd doesn't reach file-op layer for SerialDFS. /STATS, MEM /C, install, /U all work. | Likely related to pcmd's per-command `system()` model: SERDFS install via system() spawns COMMAND.COM /C → SERDFS.EXE → AX=3100h TSR. The TSR's INT 2Fh hook is in place (multiplex /STATS works), but maybe the CDS network-flag bit on X: that DOS uses to decide whether to dispatch INT 2Fh AH=11h gets cleared on COMMAND.COM /C exit. Workaround: for SerialDFS-specific testing, keep using 86box-cmd with multi-line BAT (e2e_tsr.sh: 9/9). pcmd is suitable for non-redirector DOS work and for D7-style bisects where TSR is known-loaded but not necessarily dispatching. |
| D5 | 86Box's serial passthrough mode opens a host PTY pair (`openpty()`) and writes "Slave side is /dev/pts/N" to its log; the configured `tcp_server` mode is silently ignored in the v5.3 b8200 build we ship. | Bridge layer rewritten as PTY discovery + intermediate raw-PTY pair. | `linux/ptybridge/86box_pty_bridge.py` polls the log for the slave path, opens it raw, creates an intermediate raw PTY symlinked to `/tmp/linux-com1`, throttles host→86Box writes to ~1 ms/byte (line rate at 9600 baud) to avoid UART RX overrun. |
| D2 | `inthandler` unload check used a hardcoded `+2` byte offset to find the `'SDFS'` signature, but the Watcom `__interrupt __far` prologue is ~24 bytes (signature is at `+24`). `/U` always reported "another TSR hooked its interrupt handler." | Unload path now scans the first 32 bytes of `inthandler` for `'SDFS'` (mirrors `patchhandlerds`). `/U` succeeds and prints the success message. | Fixed in this phase. |
| D3 | TSR install needed the DS/SS segment switch reduced to DS-only. The original code did `pop ss` after copying DGROUP to `newdataseg`, which crashed DOSBox-X on the next `INT 21h`. SS now stays on the original transient stack (which is abandoned anyway when the TSR exits via `INT 21h AX=3100h`); only DS is redirected to `newdataseg`. The resident handler uses the interrupted program's SS:SP at runtime, so this is fine. | Fixed in this phase. TSR installs cleanly. |
